# Manhole Card Official Audit

Use this skill only for this repository's manhole-card data workflow.

## Core Rule

Do not automatically apply search results to `data/locations.json`.

Treat search output as candidates. Apply only entries that have an official source and a card-level mapping that is clear enough to review.

## Source Priority

Use sources in this order:

1. Municipality, prefecture, or official public body pages
2. Official event or campaign pages
3. GKP pages
4. Existing repository data

Do not use fan sites, map sites, card collection sites, blogs, or aggregator search pages as source authority. They may be used only as hints for finding an official source.

Known non-authoritative examples:

- `ekikaramanhole.whitebeach.org`
- card collector pages
- general map/listing sites

## Official Design Names

`officialDesignNames` is an alias-like field for official design/topic names.

Use it for:

- Search matching in the app
- Matching the right section of official pages that list multiple cards
- Distinguishing cards whose `cardName` is only municipality plus series code

Do not use it to replace `cardName`.

Accept a name when:

- It appears on an official source page
- It identifies the card or design more specifically than the municipality name
- It is not merely the municipality or prefecture name
- It can be mapped to one card, or to a clearly documented set of cards

Hold or reject when:

- The same page lists many cards and the candidate cannot be mapped to one card
- The candidate belongs to a different municipality or prefecture-level card
- It is only a generic phrase such as "original design manhole"
- It is a source page title, navigation label, or event boilerplate rather than a design name

## English Versions

Use these statuses:

- `available`: English version is distributed or can be requested now
- `out_of_stock`: English version exists but current source says no stock or ended distribution
- `event_only`: English version is distributed only at events
- `unknown`: English version is mentioned but availability is unclear

Set `englishVersionDistributionPlaces` only when the English-version place differs from the regular distribution place.

Do not add regular distribution places to `englishVersionDistributionPlaces` for `event_only` or `out_of_stock`. That can imply the English card is available there.

## Common False Positives

Avoid these mistakes:

- Do not map a city page to a prefecture-level card just because the search query used the prefecture name.
- Do not map all names from a multi-card page to every card on that page.
- Do not treat "English version out of stock" as currently available.
- Do not accept search snippets without opening or verifying the official page.

Example: Hiroshima prefecture-level cards must not receive Hiroshima City page candidates.

## Recommended Manual Workflow

1. Inspect current repo state and avoid overwriting unrelated user changes.
2. Run data validation before and after edits when practical:
   `npm run validate:data`
3. For known official URLs, run:
   `node scripts/list-official-design-name-candidates.js`
4. For GKP-only cards, run only when the user asks for search:
   `node scripts/search-official-design-name-candidates.js`
5. Review CSV output as candidates, not as accepted data.
6. Classify rows as:
   - adopt
   - hold
   - reject
7. For adopted rows, update the deterministic source of truth:
   - `scripts/normalize-source-links.js` for verified recurring fixes
   - `data/locations.json` as generated/current data
   - `SPEC.md` when schema semantics change
   - `scripts/validate-data.js` when validation rules change
8. Run checks:
   - `npm run validate:data`
   - `node --check app.js`
   - `node --check scripts/import-gkp-data.js`
   - `node --check scripts/normalize-source-links.js`
   - `node --check scripts/validate-data.js`

## GitHub Action Policy

Do not recommend running Codex or AI search automatically from cron.

GitHub Actions may recommend manual Codex review by reporting:

- GKP-only cards
- official source candidates
- unreviewed official design-name candidates
- possible English-version mentions

The Action should not automatically edit card data from search results.

## Review Output Format

When reporting audit results, include:

- Adopted candidates with reason and official URL
- Held candidates with the missing evidence
- Rejected candidates with reason
- Any script or workflow changes needed to keep future updates stable

Keep the explanation card-level. If multiple distribution places exist, reason by card and by distribution place when relevant.
